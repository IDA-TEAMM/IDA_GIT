`src/ida_topics_yeni/ida_topics/` has 11 nodes, launched in this order by `sistem_baslat.sh`: gps_imu_driver → oakd_driver → livox_driver → sensor_node → perception_node → MAVROS → decision_node → control_node → telemetri_node → kamera_kayit_node → local_map_node.

**Written new in the prior session (2026-07):**
- `gps_imu_driver_node.py` — NMEA/UART driver for SparkFun RTK F9P, converts DDMM.MMMMM→decimal degrees, publishes `/gps/fix` (NavSatFix), parses $GNGGA/$GNRMC, exposes RTK fix quality (0=none,1=GPS,4/5=RTK).
- `oakd_driver_node.py` — DepthAI SDK driver for OAK-D Lite, publishes `/camera/image_raw` (bgr8, 640x480@30fps).
- `livox_driver_node.py` — UDP driver for Livox Mid-360, publishes `/lidar/points` (PointCloud2) and `/lidar/scan` (LaserScan, 2D slice z=0±0.3m). See [[feedback-ida-critical-rules]] for the `host_ip` gotcha.
- `control_node.py` — MAVROS-integrated control: RC kill-switch (channel 8/idx7 <1500 → emergency stop+DISARM), manual override (channel 5/idx4 >1700 → MANUAL mode), 2s watchdog timeout. Subscribes `/cmd_vel`, `/mavros/state`, `/mavros/rc/in`; publishes `/mavros/setpoint_velocity/cmd_vel`, `/mavros/rc/override`, `/control/status`, `/diagnostics`.
- `telemetri_node.py` — required by competition spec (şartname). 1Hz CSV log (lat,lon,hiz,roll,pitch,heading,hiz_setpoint,yon_setpoint,zaman) to `/tmp/telemetri/telemetri_YYYYMMDD_HHMMSS.csv`.
- `kamera_kayit_node.py` — required by spec. MP4 recording with YOLO bbox overlay (orange/yellow buoy boxes), 10fps 640x480, to `/tmp/kamera/kamera_YYYYMMDD_HHMMSS.mp4`.
- `local_map_node.py` — required by spec. LiDAR-based 2D OccupancyGrid (200x200 cells, 0.1m/cell = 20x20m area, ray-tracing for free space), publishes `/local_map` at 1Hz, saves PGM+YAML to `/tmp/local_map/` every 10s.

**Updated, not rewritten:**
- `decision_node.py` — waypoint nav + cascade PID (outer heading→yaw_rate ~10Hz, inner yaw_rate→thrust ~50-100Hz) + obstacle avoidance. Prior session's change was the QoS fix (see [[feedback-ida-critical-rules]]). Position source: `/mavros/local_position/odom`. On 2026-07-12, found and fixed a real bug via CLAUDE.md architecture review: it was still publishing `cmd_vel` to the Gazebo-only topic `/model/Girdap/cmd_vel` instead of `/cmd_vel`, so `control_node.py` (which subscribes to `/cmd_vel`) never received decision_node's commands — real-hardware control was silently broken. Now publishes to `/cmd_vel`.

`perception_node.py` — YOLO11 Nano train7 mAP50=0.984 + HSV buoy detection. **Correction (2026-07-13): actual current thresholds are orange H=10-34, yellow H=35-55 (clean, non-overlapping, matches CLAUDE.md) — an earlier version of this memory wrongly said H=20-49/H=37-80 (overlapping); don't trust that old note.** Publishes `/perception/orange_buoys`, `/perception/yellow_buoys`, `/perception/buoy_target`, `/perception/objects`.

  - **Found and fixed a real color-channel-swap bug in `perception_node.py` (commit `6aa9dd5`):** it unconditionally applied `cv2.COLOR_RGB2BGR` to every incoming frame, a leftover from when Gazebo published `rgb8`. But `oakd_driver_node.py` publishes real hardware frames as `encoding='bgr8'` (DepthAI's `getCvFrame()` is already BGR) — applying the RGB2BGR swap a second time flips R and B, corrupting `get_color()`'s HSV hue calculation. Synthetic-image test proved it: a true orange ROI classified as `'other'` before the fix, correctly as `'orange'` after. Fix mirrors the existing (already-correct) `msg.encoding`-aware pattern in `kamera_kayit_node.py`. **This means buoy color detection was very likely completely broken on real hardware before this fix** (previous real-camera tests never had buoys in frame, so this never triggered) — worth a fresh real-camera buoy test next time hardware is available, don't just trust the old "orange_buoys correctly empty" result as proof detection worked.
