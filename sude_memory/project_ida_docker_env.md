All ROS2 work for IDA/Girdap USV actually runs inside a Docker container, not on the bare host:

- Container name: `ros2_final`. As of end-of-day 2026-07-13 it runs image `ros2_harmonic_final5` (20.8GB). Chain: `ros2_harmonic_final2` (original) → `ros2_harmonic_final3` (2026-07-13, added `-v /dev:/dev`) → `ros2_harmonic_final4` (2026-07-13, committed after MAVROS/DepthAI/ultralytics/numpy-pin installs) → `ros2_harmonic_final5` (2026-07-13, committed after the colcon-build fix — setuptools/packaging version pins). `final2` and `final4` still exist as images (harmless, small marginal size due to layer sharing) but are superseded; `final3` was deleted 2026-07-13 once nothing referenced it anymore. `~/Masaüstü/ros2_baslat_v5.sh` (the from-scratch container launcher) was updated to reference `final5` — **always update that script's image tag to match whatever's newest here**, or a from-scratch relaunch will silently go back to an older, less-capable image.
- Volume mount: `/home/sudenaz/ros2_ws` (host, this repo) ↔ `/root/ros2_ws` (container), auto-synced.
- Contains: ROS2 Humble, Gazebo Harmonic, robot_localization, nav2_costmap_2d, OpenCV, YOLO11-related tooling.
- Run the node-launch script inside it: `docker exec -it ros2_final bash -c "bash /root/ros2_ws/src/ida_topics_yeni/sistem_baslat.sh"`.

**2026-07-13 container recreation (why, and what changed):**
- Root cause: OAK-D Lite camera (DepthAI/Movidius) re-enumerates on USB (new bus device number) every time it boots/connects. The container's `/dev` was a static snapshot taken at container start, not a live view of the host's `/dev` — so DepthAI always failed with `X_LINK_DEVICE_NOT_FOUND` after the boot handshake, no matter how many times the container was restarted (restart just captures a new snapshot, doesn't track live re-enumeration mid-connection-attempt).
- Fix: committed the running container to a new image (`docker commit ros2_final ros2_harmonic_final3`, preserves everything incl. MAVROS/DepthAI installs), stopped+renamed the old container to `ros2_final_old_backup` (kept, NOT removed — currently stopped, safe to `docker rm` once confirmed nothing else is needed from it), then created a new `ros2_final` container from the committed image with an added **`-v /dev:/dev` live bind mount** (all other run flags — `--privileged --network host --ipc private`, the NVIDIA GL lib bind-mounts, X11 socket, env vars — were preserved identically, see `docker inspect` output from that session if need to reconstruct again).
- Verified after recreation: `ls /dev/bus/usb/...` now shows identical, live-updating contents on host vs container. OAK-D connects successfully after this fix.
- **How to apply:** if this container ever needs recreating again (e.g. another USB device with re-enumeration issues), reuse the same `docker run` flags, always including `-v /dev:/dev`. (`ros2_final_old_backup` — the pre-recreation container this note originally warned about — was deleted 2026-07-13 after confirming it held nothing unique, see the disk-cleanup note below; this pattern of "commit first, verify, then remove the old container" is the safe procedure to repeat.)

**Real-hardware driver fixes made 2026-07-13 (see [[feedback-ida-critical-rules]] for the condensed gotcha list):**
- `oakd_driver_node.py` needs **DepthAI v2.x**, not v3 — the code uses v2 API (`pipeline.createColorCamera()`, `pipeline.createXLinkOut()`). v3 (installed by default via `pip3 install depthai`) has a totally different API and throws `AttributeError`. Installed `depthai==2.32.0.0` specifically. Also required a udev rule on the HOST for Movidius USB access: `/etc/udev/rules.d/80-movidius.rules` containing `SUBSYSTEM=="usb", ATTRS{idVendor}=="03e7", MODE="0666"`.
- `livox_driver_node.py` had two wrong hardcoded defaults, both fixed in source: device IP was `192.168.1.100`, real device (confirmed via `tcpdump`) is `192.168.117.100`; data port was `56100`, real Mid-360 point-cloud stream goes to host port `56301` (IMU data separately to `56401`, not currently consumed by this node).
- Livox requires the HOST's ethernet interface (`enp3s0` on this machine) to have a static IP on the same subnet, e.g. `192.168.117.50/24` — configured via `nmcli connection modify "Kablolu bağlantı 1" ipv4.addresses 192.168.117.50/24 ipv4.method manual` (persists across reboots since it modifies the saved NM connection profile, not just a transient `ip addr add`). DHCP will never succeed on this link since the Livox doesn't run a DHCP server.

**`colcon build` fixed 2026-07-13 (after the `ros2_harmonic_final4` commit, superseded by `ros2_harmonic_final5`):** `colcon build --packages-select ida_topics` alone fails — colcon discovers the whole vendored ROS2 source tree under `src/` and tries to build `rmw_cyclonedds_cpp`/`test_msgs` from source transitively (they're real deps of `rmw_implementation`/`rcl` respectively, already satisfied by the apt-installed `/opt/ros/humble`, but colcon prefers workspace-local source over the underlay when a same-named package exists in `src/`). **Fix: always add `--paths src/ida_topics_yeni`** to restrict package discovery so colcon never sees the vendored tree and falls back to the underlay for all deps — see CLAUDE.md's Komutlar section (this is now documented there as the canonical command). Also had to fix a second, unrelated issue along the way: `pip3 install ultralytics` (from the perception_node work) had bumped `setuptools` to 83.0.0, breaking `colcon-core` (which needs `<80`) — downgraded to `setuptools<80` (landed on 79.0.1) — and separately the system's old `packaging==21.3` didn't support a kwarg (`strip_trailing_zero`) that this setuptools version's internals call, so `packaging` was upgraded to latest (26.2) too. All three fixes (colcon-core-compatible setuptools, current packaging, the now-working `entry_points` for `sensor_node`/`decision_node`/`control_node`/`perception_node` via `ros2 run ida_topics <node>`, confirmed working) are committed into **`ros2_harmonic_final4` → `ros2_harmonic_final5`** (the currently-running container was re-committed after this fix; use `ros2_harmonic_final5` as the current reference image going forward, update `ros2_baslat_v5.sh` on Desktop too if this changes again).

**Disk space crisis 2026-07-13 (host root partition, `/dev/nvme0n1p6`, 96GB total):** hit 100% full mid-session while trying to `docker commit` (Docker images/containers had accumulated ~50GB+ of old simulation-era images and stopped containers over months). Root causes found and fixed:
- `/var/spool/abrt/` (Fedora crash-dump collector) had ~4.8GB of crash reports going back to March 2026 — safe to delete (`sudo find /var/spool/abrt/ -mindepth 1 -maxdepth 1 -exec rm -rf {} +`), not project data.
- Removed (confirmed genuinely safe first, by inspecting contents): the `ros2_final_old_backup` container (2026-07-12's pre-recreation backup, verified empty of unique data), the `ros_yolo` container (only had a generic pretrained `yolov8n.pt`, no custom data), and the by-then-superseded `ros2_harmonic_final3` image.
- **Important discovery while triaging what NOT to delete:** the container named `distracted_galileo` (Docker's auto-generated random name — clearly an accidental/forgotten container, stopped 2 months) turned out to contain the **actual YOLO training workspace**: `/workspace/images`, `/workspace/labels`, `/workspace/data.yaml`, and `/workspace/runs/detect/train3` through `train9` (including `train7/weights/best.pt` — the exact model referenced everywhere else as "YOLO11 Nano train7 mAP50=0.984"). This was its **only copy** — nowhere in git, nowhere on Desktop before this. Backed up to `~/Masaüstü/IDA_YAZILIM/yolo_egitim_yedek_20260713/workspace/` (982MB) before leaving the container itself alone (container NOT deleted, still exists stopped — the backup is a copy, not a replacement, in case there's more relevant state in the container itself like exact package versions used for training).
- **Lesson: before deleting ANY stopped container in this environment, always inspect its contents first** (`docker start <name>`, `docker exec ... find /workspace /root -maxdepth 2`, then `docker stop` again) — auto-generated container names are not a signal of unimportance, this project's most valuable ML artifact was sitting in one.
- **Follow-up cleanup same day, after applying that lesson:** inspected every remaining stopped container before removing anything. `ros_yolo_gui` turned out to hold the exact same YOLO training workspace as `distracted_galileo` (verified via `md5sum` on `train7/weights/best.pt` — identical) — pure duplicate, already covered by the `yolo_egitim_yedek_20260713` backup. `rqt_graph_temp` held an old 4-node (`control_node`/`decision_node`/`perception_node`/`sensor_node`) snapshot of `ida_topics`, matching an early project stage already preserved in git history (commit `5dede09` era). `ros2_sim`, `ros2_gazebo`, `ros2_sim_gpu`, `ros2_gazebo2` only had generic Gazebo/ROS cache dirs. All 6 removed, plus the by-then-superseded `ros2_harmonic_final4` image (kept as backup briefly after the `final5` commit, no longer needed once `final5` was confirmed working) — freed the root partition from 88GB down to 69GB used (73%, ~27GB free), well out of the danger zone it was in earlier that day.
- **Final cleanup pass, same day:** inspected the remaining old images too (`ros2_parkur_calisiyor`, `ros2_harmonic_final2`, `ros2_harmonic_final`, `ros2_gazebo_sensors`, `ros2_gazebo_final`, `ros2_gazebo_image`, `osrf/ros:humble-desktop`) by running each with `docker run --rm <image> find /root /workspace /home -maxdepth 2` — all had only standard Gazebo/ROS cache dirs, `ros2_parkur_calisiyor`'s `/root/ros2_ws` was confirmed empty (just a leftover bind-mount point, 0 bytes). All 7 removed. **`ros2_harmonic_final5` is now the only image on this machine** — clean slate. (Disk size didn't drop further from this pass since these shared underlying layers with the `final*` lineage already kept — the win was tidiness, the earlier container/final4 cleanup was where the actual ~19GB got reclaimed.)
- Also worth knowing for next time this happens: `docker image prune -f` / `docker system prune` can themselves fail with "no space left on device" once the disk is truly at 0 (Docker can't write its own metadata DB) — free some non-Docker space first (abrt, journal via `journalctl --vacuum-size=`, or similar) to get the daemon writable again before Docker-level cleanup will work.

**How to apply:**
- `docker stop ros2_final` is safe.
- `docker rm ros2_final` (or `distracted_galileo`, which still holds the only in-container copy of the YOLO training runs even though it's now backed up to Desktop too) needs a confirm-first, inspect-first approach — see the disk-cleanup note above for the procedure that's proven safe in this environment.
- Files under `/root/ros2_ws/src/ida_topics_yeni/` inside the container are the same files as this repo (volume mount) — editing here is sufficient, no need to also edit inside the container.
- `sudo` on the host requires a password (no passwordless sudo configured) — the user provided it once in-session on 2026-07-13 for udev/network setup; it was used only for that session's commands and was NOT saved anywhere. If sudo is needed again in a future session, ask the user for the password again (don't assume it's remembered/available).
